## Usage

Wrap your content with it and the beams run behind. Each beam's length, speed, width and brightness
come from a hash of its index, so no two are quite alike and the values match between server and
client (no hydration warnings). About a third of them move fast and bright like real meteors; the
rest drift slowly.

It runs on CSS keyframes alone, with no JavaScript at runtime and no resize listeners. The lanes are
a fixed set, clipped to whatever width fits and centered, so the grid crops evenly on both edges
instead of cutting off on one side. Reduced motion hides the beams.

## Changelog

### 2026-06-04

- Dropped `getBoundingClientRect` and the resize listener. The lanes are now a fixed set of
  fixed-width columns clipped to whatever width fits, so the line count stays responsive with no
  layout reads. Tradeoff: it renders a fixed number of lanes (extras are clipped) rather than
  computing the exact count for the width.
- Centered the lane field so it crops evenly on both edges instead of cutting off on the right.
- Beams vary per beam (length, speed, width, brightness) from a deterministic hash of their index,
  so no two match and server/client stay in sync. About a third now run fast and bright.
- Beams no longer peek at their resting spot before their delay (opacity is in the keyframe with
  `backwards` fill mode).
